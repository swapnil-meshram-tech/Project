Simple breakdown — **which check, on which routes**, and why.

## The three layers of checking

| Layer | What it checks | Cost |
|---|---|---|
| **Token verify** (`verifyAccessToken` / `verifyRefreshToken`) | Is the JWT signature valid? Not expired? Has `jti`? | Cheap — no DB call |
| **Active session** (`verifyActiveSession`) | Does the DB session exist, match hashed token, not revoked, not expired? Is the user active? | Expensive — DB call |
| **Role/permission** (`authorize(...)`) | Does this user have permission for this specific action? | Cheap — just checks `req.user.role` |

## Simple rule: match the check to what the route actually needs

**Routes that just need "is this a logged-in user" (most routes)**
→ Token verify + Active session (full check)

```
/profile        → verifyAccessToken + verifyActiveSession
/logout         → verifyAccessToken + verifyActiveSession
/update-account → verifyAccessToken + verifyActiveSession
/change-password → verifyAccessToken + verifyActiveSession
```

Why full check here: these routes change or reveal user data — you want to be sure the session isn't stolen/revoked, not just that the JWT signature is technically valid.

**The one route that uses the refresh token instead**
```
/refresh-token → verifyRefreshToken + verifyActiveSession
```
Same full check, just via the refresh token (cookie) instead of access token (header), because that's the credential this route receives.

**Route that needs NO session at all**
```
/forgot-password → no auth middleware
/reset-password  → verified by reset token (emailed link), not session
/login           → no auth middleware
/register        → no auth middleware
```
Makes sense — user has no valid session yet, that's *why* they're on this route.

**Routes that need role on top of session**
```
/admin/users     → verifyAccessToken + verifyActiveSession + authorize('admin')
/delete-user/:id → verifyAccessToken + verifyActiveSession + authorize('admin')
```

## Best practice — the actual rule of thumb

> **If the route touches user data or performs an action on behalf of a user → full session check (token + DB validation).**
> **If the route is public or the user isn't authenticated yet → no session check, use a different mechanism (reset token, credentials).**
> **If the route also needs a specific privilege level → add `authorize()` after the session check, never before.**

Order always matters: `verifyToken → verifyActiveSession → authorize` — never skip the DB session check just because the JWT is valid, since that's exactly what lets you catch revoked/stolen tokens.
